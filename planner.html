# overview.tf

프로젝트 개요

Terraform을 이용해 다음 AWS 리소스를 자동으로 프로비저닝하며, 프로덕션 레벨의 고가용성 및 보안을 갖춘 완전 관리형 환경을 제공합니다.

네트워크

VPC, 공개/개인 서브넷, NAT 게이트웨이

컨테이너

ECR(레지스트리), ECS Fargate(오케스트레이션)

데이터베이스

RDS MySQL 8.0 (Multi-AZ 고가용성)

로드밸런싱

Application Load Balancer (ALB)

보안

Security Groups, WAF, Secrets Manager

모니터링 / DNS

CloudWatch(로그·메트릭), Route53(도메인 관리)

# compute.tf

컴퓨팅 & 데이터베이스

컨테이너 레지스트리 (ECR)

myapp-backend(Spring Boot), myapp-frontend(React/Nginx) 두 리포지토리를 운영하며, 이미지 스캔 자동화(보안 취약점 검사)를 적용했습니다.

ECS Fargate 클러스터

클러스터명 myapp-cluster · Container Insights 활성화 · 서버리스 컨테이너 오케스트레이션 · 자동 스케일링 지원

Spring Boot 애플리케이션, 포트 8080

Frontend 태스크

# network.tf

로드밸런싱 & 보안

ALB / Target Group

타겟 그룹설명포트프로토콜

Security Groups

Security Group인바운드아웃바운드
ALB80/443 (전체 CIDR)모든 트래픽
ECS8080 / 443 (ALB에서)모든 트래픽
RDS3306 (ECS SG에서)모든 트래픽

Secrets Manager / IAM / WAF

  • DB Password, JWT Secret을 Secrets Manager에서 암호화 관리
  • ECS Execution Role(ECR·CloudWatch), ECS Task Role(RDS·Secrets Manager), RDS Monitoring Role을 각각 최소 권한으로 분리
  • AWS WAF(선택적): DDoS 방어, SQL Injection·XSS 방어, Rate limiting, 지리적 차단

모니터링 & 로깅

CloudWatch 로그 그룹 /ecs/myapp-backend, /ecs/myapp-frontend, /rds/mysql/myapp-mysql에서 CPU·메모리·네트워크 I/O·디스크 I/O·DB 쿼리 성능을 수집하고, Container Insights로 ECS 클러스터·컨테이너별 리소스 사용량을 모니터링합니다.

# deploy.tf

배포 & 프로비저닝

terraform · provisioning
# 1단계: 초기화 cd Planner-aws terraform init # 2단계: 실행 계획 검토 terraform plan -out=tfplan # 3단계: 인프라 생성 terraform apply tfplan # 4단계: 출력값 확인 terraform output # 삭제(정리) terraform destroy
출력값설명
vpc_idVPC ID
alb_dns_nameALB DNS 이름 (애플리케이션 접근 주소)
ecs_cluster_nameECS 클러스터 이름
rds_endpointRDS 엔드포인트 (DB 연결 주소)
db_secret_arn / jwt_secret_arnSecrets Manager ARN
backend/frontend_ecr_repository_urlECR 레포지토리 URL
docker · build & push
# 백엔드 이미지 빌드 & 푸시 ./gradlew build docker build -t planner-backend:latest . aws ecr get-login-password --region ap-northeast-2 | docker login --username AWS --password-stdin <ACCOUNT_ID>.dkr.ecr.ap-northeast-2.amazonaws.com docker tag planner-backend:latest <ACCOUNT_ID>.dkr.ecr.ap-northeast-2.amazonaws.com/myapp-backend:latest docker push <ACCOUNT_ID>.dkr.ecr.ap-northeast-2.amazonaws.com/myapp-backend:latest # 프론트엔드도 동일한 방식으로 빌드 → 태그 → 푸시

비용 최적화 팁

  • RDS 인스턴스 타입을 프로덕션 환경에 맞게 선택 (t3.micro → t3.small/medium)
  • ECS 태스크 CPU/메모리를 애플리케이션 요구사항에 맞게 설정
  • ECS 및 RDS Auto Scaling 정책 구성
  • 장기 운영 시 Reserved Instance로 비용 절감
  • CloudWatch 로그 보존 - 불필요한 로그는 자동 삭제 정책 설정
# troubleshooting.tf

트러블슈팅

ECS 태스크가 시작되지 않음

CloudWatch 로그 확인, ECR 이미지 존재 여부 확인, Security Group 규칙 확인

RDS 연결 오류

RDS Security Group 규칙(ECS SG에서 3306 허용) 확인, RDS 서브넷 그룹 확인, 마스터 비밀번호(Secrets Manager) 확인

ALB 헬스 체크 실패

대상 그룹 헬스 체크 경로 확인, 보안 그룹 인바운드 규칙 확인, 애플리케이션 포트 확인
